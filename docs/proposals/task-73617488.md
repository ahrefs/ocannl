# Special-Case One-Hot Virtualization for Default max_visits

**Task**: task-73617488  
**Related**: [gh-ocannl-343](gh-ocannl-343.md)  
**Date**: 2026-06-19  
**Status**: Proposed

## Goal

Make the #343 one-hot gather rewrite fire under the default `virtualize_max_visits = 1` by
teaching the virtualizer that logical one-hot selector tensors are profitable to inline.

The intended end state is:

```ocaml
let classes = TDSL.range vocab
let%op one_hot = classes = ids
let%op embedded = c * one_hot
```

lowers through the normal virtualization path, inlines the `Cmpeq` one-hot producer into the
embedding reduction, removes the one-hot producer through `cleanup_virtual_llc`, and then lets the
existing `rewrite_one_hot_reductions` pass collapse the vocabulary reduction into a guarded
`Get_dynamic`.

## Decision

Do **not** add a parallel cross-statement inlining path that rewrites through a materialized
`Get(one_hot, ...)`.

The earlier task framing proposed recognizing a materialized one-hot tensor's defining `Cmpeq`,
substituting it into a consumer reduction, and then dead-code-eliminating the now-unused
materialized producer. That would create a second inlining mechanism outside the tensor memory-mode
model:

- the tensor would remain `Never_virtual` / materialized;
- the one-hot pass would nevertheless inline it for one special consumer;
- the pass would then need its own dead-store cleanup and output-bookkeeping adjustment.

That is the wrong ownership boundary. If the tensor needs to be inlined, it should become virtual
according to the ordinary virtualizer, and the ordinary virtual cleanup should remove its producer.

## Current Behavior

#343's `rewrite_one_hot_reductions` matches reductions where the one-hot comparison is literally
present in the contribution:

```text
sum k. (k == ids[b]) * table[k, d]
```

or:

```text
sum k. where(k == ids[b], table[k, d], 0)
```

That works when the logical one-hot tensor is virtualized and inlined into the reduction body.
However, under the default `virtualize_max_visits = 1`, the producer:

```text
one_hot[b, k] = (k == ids[b])
```

reads `ids[b]`. The tracing heuristic treats that scalar computation as complex and, once the
one-hot tensor is visited more than the default cap permits, marks the one-hot tensor
`Never_virtual`. The consumer reduction then sees:

```text
sum k. Get(one_hot, [b; k]) * table[k, d]
```

The literal `Cmpeq` is gone from the contribution site, so the existing gather matcher correctly
does not fire. The current end-to-end test works around this by setting:

```ocaml
LL.virtualize_settings.max_visits <- 1000
```

That proves the #343 rewrite is sound when the producer is inlined, but it does not prove the
default pipeline finds the optimization.

## Proposed Design

Add a narrow virtualizer heuristic for logical one-hot selector producers.

During `visit_llc`, recognize tensor producers whose scalar assignment has this shape:

```text
Set {
  tn = one_hot;
  idcs = [...; Iterator k; ...];
  llsc = Cmpeq(Embed_index (Iterator k), index_expr)
}
```

with either equality operand order, and optionally the unit-affine form:

```text
Embed_index (Affine { symbols = [ (1, k) ]; offset = 0 })
```

The candidate is valid only when:

- the range side is tied to an iterator used by the producer's LHS indices;
- `index_expr` is free of that range iterator;
- the assignment is an ordinary `Set`, not `Set_from_vec`;
- the tensor is not explicitly marked materialized/non-virtual by the user or by a safety check;
- all observed setters for the candidate tensor are compatible one-hot selector setters.

For such a tensor, the visit-count heuristic should not mark it `Never_virtual` merely because it
is complex or read more than `max_visits`. Instead, leave it virtual-eligible and let the existing
`virtual_llc` / `check_and_store_virtual` path decide whether it can actually be stored and inlined.
If that later safety check rejects it, the existing `Non_virtual` path remains the fallback.

This keeps the optimization pipeline conceptually unchanged:

```text
visit_llc
-> virtual_llc
-> cleanup_virtual_llc
-> simplify_llc
-> rewrite_one_hot_reductions
-> eliminate_common_subexpressions
-> hoist_cross_statement_cse
```

The only change is that one-hot selector tensors get a targeted exemption from the generic
complexity/visit-count materialization heuristic.

## Why This Is Safe

The one-hot tensor is pure and pointwise over its own indices. Inlining it does not change ordering
or storage semantics; it only exposes the scalar equality at the consumer site.

The normal virtualizer still owns the hard safety checks:

- invalid index substitution still marks the tensor non-virtual;
- read-before-write / recurrent cases still materialize;
- explicit user materialization still wins;
- cleanup only drops producers for tensors that remain virtual-eligible.

The heuristic should therefore be implemented as "do not prematurely force `Never_virtual` for this
recognized producer", not as "force `Virtual` unconditionally".

## Non-Goals

- Do not add a cross-statement one-hot matcher that substitutes through `Get(one_hot, ...)`.
- Do not add a special DCE pass for materialized one-hot buffers.
- Do not globally raise `virtualize_max_visits`.
- Do not weaken `max_visits` for arbitrary complex pointwise computations.
- Do not add a high-level embedding/gather primitive or a data-dependent `axis_index`.
- Do not change the existing guarded `Get_dynamic` semantics from #343.

## Implementation Sketch

1. Add a small predicate near the #343 helpers, or near tracing if locality is cleaner:

   ```ocaml
   is_one_hot_selector_assignment : idcs:Indexing.axis_index array -> scalar_t -> bool
   ```

   It should share the same conceptual predicate as `match_one_hot_contribution`: equality between
   the embedded range iterator and an index expression that does not mention that iterator.

2. Extend `traced_array` bookkeeping with a conservative flag, for example:

   ```ocaml
   mutable prefers_virtual_one_hot : bool;
   mutable has_non_one_hot_setter : bool;
   ```

   The exact names are not important. The important property is that a tensor is exempt from the
   visit-count materialization rule only if every observed setter is compatible with the one-hot
   producer shape.

3. In the final `visit_llc` decision that currently marks tensors `Never_virtual` for too many
   visits, skip that mark when the tensor is a valid one-hot-virtual candidate:

   ```ocaml
   if too_many_visits && not one_hot_virtual_candidate then
     Tn.update_memory_mode tn Never_virtual ...
   ```

   Keep all existing user and safety-driven `known_non_virtual` checks.

4. Leave `virtual_llc`, `cleanup_virtual_llc`, and `rewrite_one_hot_reductions` structurally
   unchanged unless the predicate can be shared cleanly with the gather matcher.

5. Remove the `LL.virtualize_settings.max_visits <- 1000` override from the end-to-end one-hot
   embedding test, or replace it with an explicit assertion that the default setting is being used.

## Tests

Update the existing #343 test pair rather than adding a separate optimization route.

- `test/operations/test_one_hot_embedding_lookup.ml`:
  - remove the `max_visits <- 1000` override;
  - assert the optimized IR still contains `Get_dynamic`;
  - assert the vocabulary reduction loop is still gone;
  - keep the forward-equivalence, out-of-bounds, fractional-index, read-tracking, and fallback
    checks.

- `arrayjit/test/test_one_hot_gather_rewrite.ml`:
  - keep the direct structural tests for `rewrite_one_hot_reductions`;
  - optionally add a lower-level virtualizer-focused regression that builds a materialized-looking
    one-hot producer plus consumer under default settings and verifies the producer is inlined via
    the normal virtual path.

- Add one negative virtualizer regression:
  - a similar complex pointwise tensor that is **not** a one-hot selector should still obey
    `max_visits` and remain materialized when over the cap.

## Acceptance Criteria

- The one-hot embedding optimization fires with the default `virtualize_max_visits = 1`.
- The end-to-end one-hot embedding test no longer raises `max_visits`.
- The logical one-hot producer is removed by `cleanup_virtual_llc`, not by a one-hot-specific DCE
  pass.
- `rewrite_one_hot_reductions` continues to match only the inlined scalar contribution shape.
- Explicitly materialized/non-virtual tensors are not overridden by the one-hot heuristic.
- Non-one-hot complex pointwise tensors still respect the default visit-count cap.
- Existing #343 semantic checks remain intact: guarded gather, out-of-bounds returns zero, and
  fractional indices return zero.
- `OCANNL_BACKEND=sync_cc dune runtest` passes, or any unrelated failures are documented.

## Open Questions

1. Should the one-hot selector predicate live near `visit_llc` as a tracing-only helper, or should it
   be factored so `match_one_hot_contribution` and the virtualizer share one equality-recognition
   helper?

   Recommendation: share only the small "range equality against k-free index expression" helper.
   Keep producer-shape checks and reduction-contribution checks separate.

2. Should the heuristic require the one-hot tensor to have exactly one consumer reduction?

   Recommendation: no. Multiple consumers are fine as long as the producer is virtualizable under
   the ordinary inliner. The visit cap is exactly what we are overriding for this narrow, known
   profitable shape.

3. Should the heuristic set the tensor directly to `Virtual` during tracing?

   Recommendation: no. Leave it undecided but virtual-eligible. Let `check_and_store_virtual` and
   cleanup make the final memory-mode commitment.
